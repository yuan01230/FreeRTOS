# AGENTS.md

## 适用范围

- 本文件用于约束 Codex 在当前项目或当前目录树中的行为。
- 如果作为全局规则使用，放在 Codex home 目录；如果作为项目规则使用，放在项目根目录。
- 项目内更靠近当前工作目录的 `AGENTS.md` 或 `AGENTS.override.md` 可以覆盖上层规则。
- 用户在当前对话中的明确要求优先级最高。
- 本文件只保留稳定、反复需要遵守的规则；临时任务要求放在对话中说明。

## 交流偏好

- 默认使用中文回复，除非用户明确要求英文。
- 先给结论，再给必要细节。
- 对简单问题直接回答；对复杂任务先简短说明计划，再执行。
- 不要过度解释基础概念，除非用户要求教学式说明。
- 如果发现用户的需求可能有更好的实现方式，可以主动提出，但不要偏离当前目标。

## 工作方式

- 小改动可以直接执行，并在完成后说明改了什么、如何验证。
- 中大型改动、架构调整、引入新依赖、迁移数据、删除文件、批量修改前，必须先说明方案并等待用户确认。
- 修改代码前先阅读相关文件，理解现有结构和风格。
- 优先沿用项目已有框架、目录结构、命名习惯和工具链。
- 保持改动范围小而清晰，不做无关重构。
- 如果任务不明确，先根据上下文做合理假设；只有在风险较高或无法判断时才提问。

## 文档读取与写入

- 读取 Markdown、说明文档、配置文档、需求文档时，先判断是否包含中文；包含中文时优先按 UTF-8 读取。
- 在 PowerShell 中读取中文文本文件时，优先使用：
  - `Get-Content -Encoding UTF8 "C:\path\to\file.md"`
- 如果终端输出出现中文乱码，不要基于乱码内容继续分析；必须用 UTF-8 重新读取后再判断。
- 写入 Markdown、说明文档、配置文档时，默认使用 UTF-8。
- 修改已有文档时，保留原有结构和语气；只调整与任务相关的内容。
- 不要把临时分析、长篇推理、无关背景写进项目文档。
- 如果文档是对外发布、PR 描述、README、规范说明，写完后检查标题层级、列表缩进、代码块和链接是否正常。

## 安全规则

- 禁止批量删除文件或目录。
- 不要使用：
  - `del /s`
  - `rd /s`
  - `rmdir /s`
  - `Remove-Item -Recurse`
  - `rm -rf`
- 需要删除文件时，只能一次删除一个明确路径的文件。
- 正确示例：
  - `Remove-Item "C:\path\to\file.txt"`
- 如果需要批量删除文件，应停止操作，并请求用户手动删除。
- 不要执行破坏性 Git 操作，除非用户明确要求，例如：
  - `git reset --hard`
  - `git checkout -- <file>`
  - 强制覆盖用户改动
- 如果工作区已有用户改动，不要擅自回滚；应保留并绕开无关改动。

## 代码修改原则

- 优先做最小可行修改。
- 不要为了“顺手优化”改动无关文件。
- 不要随意新增依赖；确实需要时先说明理由。
- 不要大面积格式化文件，除非项目明确要求或用户要求。
- 修 bug 时先定位原因，再修改；不要盲目试错。
- 新功能优先补充或更新相关测试。
- 对共享逻辑、核心流程、数据处理、权限、安全相关代码，必须格外谨慎。

## 项目规则补充

- 如果项目提供 README、贡献指南、脚本说明、CI 配置或已有 `AGENTS.md`，先阅读与任务相关的部分。
- 优先使用项目已有包管理器和命令，例如 `npm`、`pnpm`、`yarn`、`pip`、`uv`、`cargo`、`go`、`make` 等，不要擅自切换工具链。
- 新增依赖前先确认项目已有依赖是否能解决问题。
- 涉及 monorepo 时，先定位实际子项目，不要在仓库根目录盲目运行安装、测试或格式化命令。
- 若项目有专门测试命令、构建命令、代码生成命令，应优先按项目文档执行。

## 验证要求

- 修改后尽量运行相关测试、lint、类型检查或启动检查。
- 如果无法运行验证命令，需要说明原因。
- 不要只说“应该可以”，要说明实际验证过什么。
- 如果验证失败，先报告失败原因和下一步建议，不要假装完成。
- 验证范围要与改动范围匹配：小改动运行相关测试；核心逻辑或共享模块改动需要更完整的验证。
- 不要为了通过测试而删除、跳过或弱化测试，除非用户明确要求并理解风险。

## 前端与界面

- 优先保证可用性、清晰度和响应式表现。
- 不要做花哨但影响可读性的设计。
- 移动端和桌面端都要避免文字溢出、按钮挤压、内容重叠。
- 已有设计系统时，优先遵循现有组件和视觉风格。
- 用户要求做应用或工具时，优先做可用界面，不要先做营销落地页。

## Git 与提交

- 不要自动提交，除非用户明确要求。
- 提交前先查看 diff，确认没有无关文件或敏感信息。
- 不要默认使用 `git add .`；优先按逻辑分组添加明确路径。
- 不要自动 push、force push、merge、rebase，除非用户明确要求。
- 生成 commit message 时，描述真实改动，不夸大范围。

## 输出格式

- 完成任务后，用简短中文总结：
  - 改了什么
  - 验证了什么
  - 是否还有风险或后续建议
- 如果涉及文件，给出文件路径。
- 如果是代码审查，优先列问题、风险和对应位置，不要先写泛泛总结。

## AGENTS.md 维护原则

- 当用户指出同类问题第二次出现时，优先建议把规则补充进 `AGENTS.md`。
- 新增规则要具体、可执行、可验证，避免空泛口号。
- 不把一次性偏好、临时任务、项目无关知识写进 `AGENTS.md`。
- 如果规则与项目实际命令冲突，以项目文档和用户最新指令为准，并建议修正冲突。
- 定期删除过时规则，保持文件短而有效。

## 用户偏好

- 用户偏好平衡型协作：小事自动完成，大事先确认。
- 用户重视安全、可控、不要乱删文件。
- 用户希望 Codex 主动给建议，但不要擅自扩大任务范围。
